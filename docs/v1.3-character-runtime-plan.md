# v1.3 Character Runtime Plan

> Goal: turn the current Character Runtime demo path into a dependable interaction loop while keeping the project safe, low-interruption, and honest about what is still experimental.

## Product Goal

v1.3 should make the character feel more coherent across a conversation:

- replies can carry structured emotion/action intent
- Live2D feedback reacts to that intent without breaking chat or TTS
- TTS playback and motion timing feel less random
- runtime behavior is observable and easy to turn off
- default behavior remains safe for new users

This is still an open-source MVP track, not a claim of a complete AI VTuber or full desktop agent.

## Non-Goals

- Do not enable automatic desktop observation by default.
- Do not enable tool calling or shell execution by default.
- Do not make proactive interruptions frequent or unavoidable.
- Do not promise mature long-term memory or plugin marketplace behavior.
- Do not introduce large framework/dependency changes just for runtime work.
- Do not refactor all of `app.py` or `web/chat.js` in this milestone; only do small extraction when it directly reduces risk.

## Current Foundation

Already available:

- `character_runtime.py`
  - event model
  - state snapshot
  - safe payload normalization
  - emotion/action normalization
  - low-interruption proactive gate skeleton
- Backend runtime prompt/reply path
  - opt-in structured JSON contract
  - optional `character_runtime` metadata
  - `/api/chat` and `/api/chat_stream` metadata support
- Frontend bridge
  - metadata normalization
  - `character-runtime:update` event
  - `BroadcastChannel("taffy-character-runtime")`
  - emotion/action Live2D bridge
  - debug bridge for manual checks
- Documentation
  - demo enablement
  - smoke test
  - safe local config
  - release validation notes

## Milestone Tasks

### 1. Runtime Defaults and Config Boundary

- Confirm runtime stays disabled by default.
- Keep `return_metadata=false` by default.
- Add clearer config docs for:
  - `enabled`
  - `return_metadata`
  - `demo_stable`
  - persona override fields
- Add a single runtime config summary in health/debug output if it does not expose secrets.

Acceptance:

- default install has no response-shape change
- runtime opt-in is documented in one obvious place
- unsafe config combinations produce warnings, not silent surprises
- done: protected `/api/health` and startup self-check now report only safe runtime flags

### 2. Metadata Reliability

- Tighten metadata normalization on backend and frontend.
- Ensure malformed JSON, plain text, and partial JSON never leak raw JSON to chat UI.
- Add tests for:
  - unknown emotion/action fallback
  - metadata without text
  - stream done payload consistency
  - provider failures preserving useful diagnostics

Acceptance:

- chat text remains user-readable
- runtime metadata is optional and stable
- provider errors are not misattributed to runtime
- done: backend and frontend metadata fallback checks cover malformed, unknown, and optional metadata paths

### 3. Live2D Feedback Quality

- Review current emotion/action mapping.
- Add a small mapping table document for supported model resources.
- Improve fallback feedback when motion resources are missing.
- Keep feedback subtle and non-blocking.
- Keep `speech_motion_strength` configurable and bounded.

Acceptance:

- `happy`, `surprised`, `thinking`, `annoyed`, `wave`, `shake_head` have visible or safely degraded behavior
- missing motion resources do not crash the app
- user can still chat normally if runtime bridge fails
- done: supported emotion/action values and fallback behavior are documented in `docs/character-runtime-live2d-mapping.md`

### 4. TTS Timing and Speech Loop

- Keep TTS provider selection independent from `voice_style` until voice-style TTS is implemented.
- Improve timing between:
  - reply text arrival
  - speech start
  - mouth/motion feedback
  - speech end / idle return
- Add notes for GPT-SoVITS demo use without making it the default first-run path.

Acceptance:

- browser TTS remains the easiest first-run path
- GPT-SoVITS can be used locally for demos
- TTS failure does not block text chat
- speech motion returns to idle reliably
- done: successful TTS completion now uses a short speech release path, while failures and manual stops still hard-stop animation
- done: demo docs now describe GPT-SoVITS as optional local demo setup, not the default first-run path

### 5. Debug and Validation Workflow

- Keep DevTools debug bridge developer-only.
- Add a concise runtime validation script or checklist entry for:
  - runtime disabled baseline
  - metadata opt-in
  - cross-window bridge
  - emotion/action feedback
  - TTS/text regression
- Keep release/demo docs free of private local configuration.

Acceptance:

- contributor can validate runtime without reading source code first
- smoke test results can be recorded repeatably
- public demo material does not show secrets or private desktop content
- done: validation results can be recorded with `docs/character-runtime-validation-log.md`

### 6. Low-Interruption Proactive Behavior

- Keep proactive behavior off or strongly gated by default.
- Define explicit triggers before implementation:
  - cooldown
  - recent user activity
  - recent assistant activity
  - user-configured enable switch
- Prefer local debug/demo checks before any public default behavior.

Acceptance:

- no surprise interruptions in default config
- proactive path can be disabled quickly
- runtime state explains why a proactive action did or did not happen
- done: standalone runtime proactive scheduling now requires an explicit enable switch and reports safe block reasons
- done: proactive safety boundary is documented in `docs/character-runtime-proactive-safety.md`

## Release Criteria

v1.3 can be considered ready when:

- default behavior is unchanged and safe
- runtime opt-in path works across chat, stream, frontend bridge, and model window
- Live2D feedback visibly improves for supported emotion/action cases
- TTS/text regressions are not introduced
- docs tell contributors how to enable, validate, and roll back runtime behavior
